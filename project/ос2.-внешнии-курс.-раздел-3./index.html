                    ОS2. External course. Part 3

                    Nov 22, 2025 · 11 min read

                    The third section of the course “Linux System Administrator from scratch” is presented here.

                    Objective

                    Complete the third section of the external course “Linux System Administrator from Scratch”.

                    Task

                    Tasks from the eighth, ninth, tenth, and eleventh modules, as well as tests.

                    Theoretical Introduction

                    • Module 8: Network and SSH Configuration
                    • Module 9: Package Management
                    • Module 10: Log Management
                    • Module 11: Containerization with Podman

                    Practical Tasks Execution

                    Configuring a static IP on the test server manually.

                    Writing the configuration for the eth0 interface in the /etc/network/interfaces file.

                    Checking the accessibility of external resources via ping.

                    Removing the current IP address from the interface. Starting any web server on port 80 (e.g., nginx) and checking that it works.

                    Using the dig utility to find out the IP addresses of popular services.

                    Checking if the SSH port is listening on the server.

                    Changing the SSH port and disabling password login.

                    Configuring key-based connection and authorizing.

                    Allowing SSH access only for specific users. Verifying that connection for another user does not work.

                    Configuring UFW to restrict access only to SSH.

                    Configuring UFW so that SSH access is only allowed from your local network.

                    Using fail2ban, creating a jail that will block an IP address after three successful requests to nginx.

                    Checking how the blocking works with multiple requests.

                    Finding and installing the htop utility.

                    Removing the htop utility, then reinstalling it with complete purge.

                    Performing a full system update.

                    Configuring automatic installation of security updates.

                    Removing obsolete and unused dependencies.

                    Installing a package manually via dpkg, after downloading it from the website.

                    Analyzing which dependencies will be required.

                    Viewing the contents of the main system log and authentication log using a paging utility.

                    Displaying the last 20 entries from the binary system log using journalctl.

                    Checking the existence and location of the Nginx web server error log file.

                    Studying any one line from the /var/log/auth.log file and manually identifying: the timestamp, service (process) name, and event description.

                    Checking the current status of the rsyslog and systemd-journald services using systemctl.

                    Generating a test message using the logger utility. Then finding this message first in the journalctl output, and then in the /var/log/syslog file. Using journalctl, filtering and displaying only events with error (err) severity level or more critical. Looking into the rsyslog configuration file (e.g., /etc/rsyslog.d/50-default.conf) and finding the line responsible for directing messages from the auth and authpriv facilities to the /var/log/auth.log file. Sending a message with an explicitly specified warning severity level to the system log. Verifying that it appears in the journalctl output when filtered by this level.

                    Finding all lines in the /var/log/auth.log file that mention a failed password login attempt (Failed password), ignoring case. Building a command pipeline that first finds all successful SSH connections (Accepted password) in /var/log/auth.log, and then extracts only the IP addresses of the connected clients from these lines. Compiling a list of unique IP addresses from which failed login attempts were recorded. Collecting statistics and displaying the five IP addresses from which the largest number of successful SSH logins were recorded.

                    Using journalctl, displaying all system logs from the last 15 minutes.

                    Finding all login attempts by non-existent users in the authentication log /var/log/auth.log.

                    Determining the IP address from which the largest number of failed password attempts (Failed password) were made. After identifying the most suspicious IP address from the previous step, extracting absolutely all entries related to this IP from /var/log/auth.log and saving them in a separate file incident_report.log. Calculating the SHA256 checksum for the created incident_report.log file to record its integrity and prove that it has not been modified after collection. Checking which commands were executed with superuser privileges (via sudo) by your current user.

                    Studying the existing logrotate configuration for the system package manager (apt or dpkg) in the /etc/logrotate.d/ directory. Determining how often rotation occurs, how many archival copies are stored, and whether compression is used.

                    Creating a custom configuration file /etc/logrotate.d/testapp to manage a fictional log file /var/log/testapp.log. Configuring it for daily rotation, storing four archival copies, and compressing old logs.

                    Checking the created logrotate configuration for syntax errors by performing a “dry run” in debug mode.

                    Forcing rotation for the testapp log and checking the result: ensuring that the old log was compressed and renamed, and a new empty file appeared in its place. Writing a configuration line for rsyslog that will forward absolutely all logs (.) via TCP protocol to a remote server with the address logs.example.com and the standard port 514.

                    Installing Podman and verifying the installation.

                    Starting a container.

                    Starting Nginx and verifying that it works.

                    Deploying a website in Nginx using a local folder with HTML files.

                    Running an Nginx container with memory limited to 100 MB and checking its operation.

                    Investigating the container’s behavior when the allocated memory is exhausted.

                    Creating and running a custom website based on nginx using Podman.

                    Saving the image and transferring it to another server.

                    Test Tasks Execution

                    We use the ip a add command to temporarily add an IP address to a network interface. The auto directive in the /etc/network/interfaces file tells the system which interfaces to automatically activate (bring up) during boot. To remove a specific IP address from an interface, we use the ip a del command specifying the full address with mask and the interface name.

                    Settings configured using the ip command are temporary and only last until system reboot. When statically configuring the network in the /etc/network/interfaces file, we specify the default gateway using the gateway directive in the interface configuration block.

                    We use the modern ss utility with the -t, -u, -l, -n, -p keys to get a complete list of open ports and the processes associated with them. The -p key in the ss command forces it to display the Process ID (PID) and name of the process using the socket.

                    We use the dig command to perform DNS queries and get detailed information about domain names, including their IP addresses. For quickly checking the availability of a remote TCP port, we use the netcat utility with the -v and -z keys.

                    By default, the SSH server daemon listens for incoming connections on TCP port 22. The main configuration file for the SSH server is located at /etc/ssh/sshd_config. The ssh_config file is for the client side. To temporarily stop a system service, we use the systemctl stop command.

                    To set up SSH key authentication, we copy the contents of the public key file to the authorized_keys file on the server. The PermitRootLogin no directive in the /etc/ssh/sshd_config file prohibits direct SSH connection to the server under the root account, which is an important security measure.

                    After configuring the rules, we activate the UFW firewall with the ufw enable command. This applies the rules and enables firewall autostart at system boot. The main Fail2ban jails settings, which define blocking rules, are stored in the /etc/fail2ban/jail.conf file. To remove a rule by its number from the list displayed by the ufw status numbered command, we use the ufw delete [number] command.

                    The bantime parameter in the Fail2ban jail configuration defines the duration of IP address blocking in seconds after exceeding the attempt limit. To restrict access to a port by source, we use the syntax ufw allow from [source] to any port [port]. This allows connections only from the specified subnet.

                    The apt autoremove command removes packages that were automatically installed as dependencies and are no longer needed. The target packages themselves are removed with the remove command, and their configs remain. The apt purge command removes a package along with its configuration files. Dependencies installed with it remain in the system. We regularly run apt update to update the local package database. Without this, the system will not know about new package versions.

                    The unattended-upgrades package allows us to configure automatic installation of security updates and other packages without manual intervention. After apt update, we run apt upgrade to install all available updates for installed packages. The apt full-upgrade command performs a smarter update that may remove obsolete packages or install new dependencies, which is sometimes necessary for a full system update.

                    When dependency problems occur, we use the apt --fix-broken install command to try to automatically fix broken dependencies. Forcibly removing a package on which other programs depend leads to “dependency breakage”. To filter the output of other commands and search for a specific package, we use a pipeline with grep.

                    We prefer to install .deb files via apt, as it automatically resolves and installs all dependencies. APT checks the integrity and authenticity of packages from repositories using GPG keys, which protects the system from installing modified or malicious packages. In a corporate environment, if the required package is not in the approved repositories, the correct action is to request its addition through the ticketing system, not to install it yourself from unverified sources.

                    We use logs primarily for these three purposes: to find the cause of a malfunction, to investigate the circumstances of a hack, and to identify “bottlenecks” in the system. Error 502 usually indicates a communication problem between the web server and the backend process. In modern systems, logs are stored in two main places: classic text files in /var/log/ and the centralized systemd binary journal, which we view using journalctl.

                    Typically, systemd-journald acts as the primary log collector, and then, if settings are present, forwards messages to the syslog daemon for permanent storage in familiar text files in /var/log/. The failure to start a critical system service is a significant negative event classified as an error level, not just an informational message. A facility in syslog indicates the subsystem or program that is the source of the message.

                    For uniq to correctly count duplicate lines, they must follow each other. In awk, $1, $2, $3, etc. denote the first, second, third, and subsequent fields in a line. We use a pipeline: the first grep filters lines with “error”, and the second grep with the -v key removes lines containing “healthcheck” from this result.

                    A large number of failed login attempts in a short period of time, especially for standard usernames, is a classic sign of an automated brute-force attack. Calculating the hash records the current state of the file. Commands executed via sudo are extensively logged in authentication logs.

                    The main task of logrotate is the rotation, archiving, and deletion of old log files according to a set schedule and rules. Remote log collection ensures log integrity. Grafana Loki is designed specifically for this scenario.

                    The podman info command outputs detailed information about the Podman environment: version, configuration, storage, network, etc., which helps us verify its correct setup. The key architectural difference of Podman is that it uses a daemonless architecture. The --rm key automatically removes the container immediately after it finishes its work.

                    Podman’s security for multi-user environments is due to its ability to operate in rootless mode. As already noted, Podman does not require a constantly running background daemon.

                    We use the podman run command to start a new container. The -v flag is used to mount directories or files from the host machine into the container. The combination of the -i and -t flags allows us to run the container in interactive mode with a pseudo-TTY.

                    By default, podman ps shows only running containers. To see all containers, including stopped ones, we add the -a flag. The --rm flag forces Podman to automatically remove the container immediately after it is stopped.

                    To monitor the resource consumption of running containers in real time, we use the podman stats command. The -d flag runs the container in the background. After launch, control returns to the terminal, and the container continues to run independently. After modifying systemd unit files, we run systemctl daemon-reload.

                    When a container is run as a systemd service, its logs are integrated into the general systemd journal. The --memory flag sets the maximum limit of RAM that the container can use.

                    To download Docker images from a remote registry, we use the podman pull command. The policy.json file defines the trust policy for container images. In it, we configure which registries are allowed to download images from, whether a digital signature is required for them, and which keys are trusted. The podman build command is used to build your own container image.

                    We use podman build -t myapp . to create our own image from the instructions in the Dockerfile located in the current directory. The result will be an image named myapp. If the trust policy (policy.json) for a specific registry or image specifies the signedBy directive, then Podman will check for the presence and validity of a digital signature for the image.

                    Test Scores

                    Conclusions

                    We have completed the third section of the external course “Linux System Administrator from Scratch”.
