                    ОS2. External course. Part 2

                    Nov 22, 2025 · 10 min read

                    The second section of the course “Linux System Administrator from scratch” is presented here.

                    Objective

                    Complete the second section of the external course “Linux System Administrator from Scratch”.

                    Task

                    Tasks from the fourth, fifth, sixth, and seventh modules, as well as tests.

                    Theoretical Introduction

                    • Module 4: Getting Help. Using reference systems, working with text files and logs
                    • Module 5: User and Group Management
                    • Module 6: Access Control
                    • Module 7: Process Management

                    Practical Tasks Execution

                    Using man to learn how the grep command works.

                    Finding documentation about systemctl using info.

                    Opening the local documentation about info.

                    Opening the /etc/os-release file using less and finding the distribution name.

                    Using grep to find lines containing “error” in the /var/log/syslog file.

                    Editing the SSH configuration using nano or vim and changing the PermitRootLogin parameter to no.

                    Creating a small file with a matrix of numbers of arbitrary size, and experimenting with the numbers using the awk utility — for example, calculating the sum of numbers in each even row.

                    Finding all errors in the system log from the last day.

                    Checking SSH logs and finding failed login attempts.

                    Writing a script that daily saves all errors from /var/log/syslog to a file.

                    Configuring cron so the script runs once a day.

                    Using grep to count the number of failed login attempts in the last 24 hours.

                    Creating a new user ‘ivan’ and setting a password for them.

                    Creating a new group ‘developers’. Adding user ‘ivan’ to the ‘developers’ group. Creating a group ’testers’ and changing user ‘ivan’s group membership from ‘developers’ to ’testers’. Deleting the user account ‘ivan’ along with their home directory.

                    Creating a new file somewhere. Examining the permissions it received automatically. They have these specific values because of umask.

                    For the created file, removing write permissions for the group and read permissions for others. Restoring the original permissions to the file using the numeric form.

                    Making the root user the owner of the file.

                    Taking back ownership of the file.

                    Performing several actions using sudo and finding them in the system log.

                    Using the ‘ivan’ account from previous lessons. Checking if they can execute superuser commands. If not, granting them this ability.

                    Displaying the names of all users who have the outdated SHA-512 algorithm specified for their password hashing (its prefix is $6$).

                    Creating user ‘ivan’. Setting the required password policy parameters for them using the chage command. Locking the ‘ivan’ user account.

                    Verifying that they cannot log into the system.

                    Restoring the ‘ivan’ user’s ability to log in.

                    Checking the access permissions for /etc/passwd, /home, and /var/log.

                    Creating a file in the home directory, checking its permissions, and setting umask so that new files have no permissions for the ‘others’ category.

                    Creating a file and setting its permissions to 644, so the owner can edit, and others can only read. Creating an ‘admins’ group, changing the group of the file created in the first task to this one, and granting the group write permission.

                    Allowing a specific user to modify the file without adding them to the owner’s primary group. Setting special permissions for the group so they can only execute the file. Checking and resetting all ACLs on the file.

                    Setting the SUID on an executable file so it runs with the owner’s privileges. Configuring SGID for a directory so new files inherit the group. Making it so the ‘others’ category can write to /public_folder but cannot delete others’ files.

                    Finding the process consuming the most memory and terminating it.

                    Starting a process in the background, bringing it to the foreground, suspending the process, then returning it to the background.

                    Installing and running htop, configuring the desired columns, finding a process by filter and “killing” it.

                    Starting a task with low priority — for example, with nice=15. Finding the PID of a previously started process and increasing its priority to -5. Finding the process with the lowest priority and trying to increase its priority.

                    Listing all active services and filtering them by the “Network” filter. Determining which web server is installed, checking the service status, and restarting it. After restarting, checking the log for errors.

                    Finding an unnecessary service. Ensuring that this process does not start upon system reboot. Rebooting the device and checking that the selected service is turned off.

                    Creating a simple Python script and giving this file execution rights.

                    Creating a unit file — activating and checking the status of your unit.

                    Forcibly terminating the process that was started in the previous task and checking the service status.

                    Rebooting the device and checking if the autostart of your process worked.

                    Test Tasks Execution

                    When we need to get a complete and structured description of a utility, its options, and usage examples, we refer to the built-in manual using the man command. We use the info command as an alternative to man for viewing more detailed documentation. We know that after installing a package, its additional documentation, licenses, and configuration examples are usually placed in the /usr/share/doc/package_name directory.

                    We use the cat > file.txt construct to redirect standard input into a new file. We use the >> operator when we need to append data to the end of an existing file without erasing its previous content. The > operator always creates a file anew or completely overwrites an existing one. For viewing large files, we prefer to use less because it allows us to scroll through content page by page, search text, and move freely through the file, whereas cat dumps all content at once, which is inconvenient for navigation.

                    To exit the less viewer, we simply press the q key. In the Vim editor, while in Normal mode, we press the : key to switch to Command mode.

                    We refer to the /var/log/ directory as it is the standard location for storing system and application logs. We use this command to filter and view all systemd journal entries related to the sshd service since the start of the current day. To see a limited number of recent entries, we use the -n key. The command journalctl -n 20 will output the last 20 lines of the journal.

                    To monitor a growing log file in real time, we use the tail command with the -f flag. We know that individual cron jobs for each user are stored as files in the /var/spool/cron/ directory.

                    In a cron entry, we use the asterisk symbol * to denote “every” valid value in a field (e.g., every minute, every hour). To completely clear our personal cron table, we run the crontab -r command.

                    When creating a user, we use the -m flag so the system automatically creates a default home directory for them. To completely remove a user account along with their home directory and all its contents, we use the userdel command with the -r flag.

                    We use this command to grant the ‘admin’ user superuser privileges. The -aG flag ensures the user is added to the sudo group without losing other group memberships. For editing critical system files /etc/passwd and /etc/shadow, we use the special vipw utility. It locks the file during editing, preventing simultaneous modification from elsewhere, which ensures data integrity.

                    The first character in the ls -l output indicates the file type. The letter ’d’ means it is a directory. We use the chown command in the format “chown owner:group file” to simultaneously change the owner and group of a file. We decode the numeric permission format as follows: 7 (owner: rwx), 5 (group: r-x), 4 (others: r–). This corresponds to the string rwxr-xr--.

                    All commands executed via sudo are logged in the system journal, allowing us to track who, when, and what was done with root privileges. After changing the SSH port, the daemon will run on the new port, but the firewall by default blocks all incoming connections except those allowed by rules.

                    To allow incoming connections to a non-standard SSH port, we add a rule to UFW explicitly specifying the port number and protocol: sudo ufw allow 47022/tcp. We prefer sudo because it allows granting privileged access without disclosing the root password, provides detailed logging of actions, and offers more flexible rights management through the /etc/sudoers file.

                    We understand that hashing is a one-way cryptographic function. The hashing algorithm identifier, such as $6$ (SHA-512) or $y$ (yescrypt), tells the system which method was used to generate that hash. We add a “salt” — a random string — to the password before hashing. This ensures that even identical passwords will have different hashes.

                    To check password policy settings, we use the chage command with the -l flag. The usermod -L command locks a user account by prepending a ! to the beginning of their hashed password in /etc/shadow, making it impossible to log in with that password.

                    To view a detailed list of files, including their access permissions, owner, and group, we use the ls -l command. The combination of -l and -a flags in the ls command allows us to see the permissions of all files, including hidden ones. We convert symbolic permissions to numeric: r (read) = 4, w (write) = 2, - (no execute) = 0. Sum: 4 + 2 + 0 = 6.

                    To change access permissions, we use the chmod command. In the symbolic mode of the chmod command, we use operators: + to add permissions, - to remove permissions, and = to set permissions to an exact value. To recursively apply permission changes to all files and subdirectories within a directory, we use the -R flag.

                    We set the Sticky Bit on public directories like /tmp so users can only delete their own files, even if they have write permission to the directory. SUID is a special permission that causes an executable file to run with the privileges of its owner, not the user who launched it. Such a parameter does not exist.

                    To get a detailed list of all processes in the system with information about the user, PID, CPU, and memory usage, we use the ps aux command. When a process does not respond to other signals, we use the kill -9 command, which sends the SIGKILL signal. We suspend a process running in the terminal by pressing Ctrl+Z. Then, to resume its execution in the background, we enter the bg %1 command.

                    The nice value determines the scheduler’s priority. The lower the value, the higher the process priority. Thus, a value of -20 is the highest priority, and 19 is the lowest. To change the priority of an already running process, we use the renice command. In a systemd unit file, we use the Nice= directive to set the nice value for all processes of this service upon their startup.

                    To check the state, activity, and recent logs of a system service, we use the systemctl status service_name command. In a systemd timer, we use the OnCalendar= directive with the daily parameter to schedule a daily run at midnight. To have a service start automatically at system boot, we “enable” it using the systemctl enable command. This creates the necessary symbolic links.

                    We use the Restart=always directive in a systemd unit file when we need the service to automatically restart regardless of how it ended — normally, with an error, or was forcibly stopped. This ensures maximum service fault tolerance. To monitor the log of a specific service in real time, we use the journalctl -u servicename -f command. Before running a new service, we always check its unit file for syntax errors using the systemd-analyze verify command.

                    We have completed the second section of the external course “Linux System Administrator from Scratch”.
